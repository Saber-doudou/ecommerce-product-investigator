# 数据采集降级指南

当首选数据源不可用时，按降级链切换到备选方案，确保调研不中断。

---

## 降级链（通用）

```
API 数据 ──→ CDP 浏览器 ──→ Web Search ──→ 用户手动提供
 (最快最稳)    (数据最全)      (通用兜底)       (最后手段)
```

每次降级时必须告知用户：
1. 降级了哪个数据源
2. 降级原因（一句话）
3. 数据精度可能的变化

---

## 精度标注体系

降级采集的数据在报告中必须标注来源和精度等级：

| 标注 | 含义 | 使用场景 |
|:---:|:---|:---|
| 🟢 高 | API/已登录页面直读 | 买手 API、CDP 登录态 |
| 🟡 中 | 第三方站补充 | ZOL、什么值得买 |
| 🟠 低 | 图片多模态读取 | 参数为图片形式 |
| 🔴 手动 | 用户手动提供 | API 和 CDP 均失败 |
| ❓ 未检索到 | 所有方法均未获取到 | 明确标注，不猜测 |

**标注格式示例**：
```markdown
| 数据项 | 来源 | 精度 | 备注 |
|--------|------|------|----------|
| 竞品价格 | 买手 API | 🟢 高 | 实时数据 |
| BSR 排名 | Web Search | 🟠 低 | 非实时，可能滞后 |
| Review 分布 | CDP 采集 | 🟡 中 | 页面快照，可能有延迟 |
```

## 降级场景与处理

### 场景1：API 不可用（买手引擎失败）

| 错误 | 原因 | 处理 |
|:---|:---|:---|
| API key 无效 | 环境变量未设置或过期 | 提示用户设置 `MAISHOU_API_KEY`，同时降级到 CDP/Search |
| 超时 | 网络问题或服务端慢 | 重试1次（间隔3秒），仍失败则降级 |
| 返回空结果 | 关键词不匹配 | 尝试换关键词（英文化/简化），仍无结果则降级 |
| 429 Too Many Requests | 请求频率过高 | 等待30秒后重试1次，仍失败则降级 |

> 💡 **核心原则**：买手引擎是加速器，不是必选项。失败了不影响最终结果，CDP 引擎能兜底。

### 场景2：CDP 连接失败

| 错误 | 原因 | 处理 |
|:---|:---|:---|
| Connection refused | 浏览器未启动调试模式 | 提示用户启动（参考 `cdp-setup.md`），不等待，直接降级到 Search |
| 9222端口无响应 | 端口被占用或非浏览器占用 | 自动尝试9223/9224端口；全部不可用时提示用户 |
| `agent-browser` 连接超时 | 浏览器未用 `--remote-debugging-port` 启动 | 等待2秒后重试1次，仍失败则降级 |
| 页面加载超时 (>30s) | 网络慢或页面复杂 | 重试1次（`wait --load networkidle`），仍失败则降级到 Search |
| 快照返回空白 | 页面未完整加载 | 重新打开URL + 等待；尝试切换移动端UA |

### 场景3：登录墙探测异常

| 错误 | 原因 | 处理 |
|:---|:---|:---|
| 看到登录弹窗 | 用户未登录 | 提醒用户完成登录，等待30秒后重新探测；或关闭弹窗后重试 |
| 页面空白/验证页 | 触发反爬验证 | 等待5秒后重新snapshot；尝试添加Referer头或换User-Agent |
| 重定向到登录页 | Cookie过期 | 不重试，告知用户需要先登录，给出登录步骤 |

### 场景4：数据提取失败

| 错误 | 原因 | 降级方案 |
|:---|:---|:---|
| 规格参数读取为空 | 页面异步渲染 / 未展开tab | 点击"规格参数"tab重试1次 → Route D（ZOL参数页）→ Route A（API）→ Route C（图片多模态）→ **browser-use兜底** |
| 评价数据缺失 | 默认只显示好评 | 切换到"全部评价"筛选重试 → 第三方站评价（ZOL/什么值得买）→ **browser-use兜底** |
| 价格显示"登录查看" | 登录墙 | 用买手引擎的价格数据替代，或询问用户登录 |
| 图片URL无法下载 | 防盗链 / 签名过期 | 换User-Agent重试 → 标注"未检索到"，跳过该项 |
| BSR排名数据缺失（跨境） | 页面结构变化 | 用 Keepa / Jungle Scout API 补充 → Web Search 估算 |

### 场景5：Web Search 结果不足

| 情况 | 处理 |
|:---|:---|
| 搜索无结果 | 换关键词（同义词/品类上位词），最多尝试3组 |
| 结果相关度低 | 用 `site:` 限定搜索范围，如 `site:amazon.com "keyword" review` |
| 信息过时 | 标注数据时效，提示用户"数据可能非最新" |
| 严重不足 | 明确告知用户数据缺口，建议用户手动补充 |

## browser-use 兜底

当所有常规方法（Route B/D/A/C）都失败时，启用 browser-use 作为**最后手段**：

```bash
# 安装（仅需一次）
pip install browser-use
playwright install chromium

# 使用示例：让AI自己理解页面并提取
python -c "
from browser_use import Agent
import asyncio

async def main():
    agent = Agent(
        task='从这个商品页提取所有规格参数，输出JSON格式',
        llm_model='YOUR_LLM_MODEL',  # 例如 gpt-4o / deepseek / glm 等
        headless=True
    )
    result = await agent.run()
    print(result)

asyncio.run(main())
"
```

> ⚠️ **browser-use 使用注意**：
> - **速度慢**：每次操作需LLM推理，约2-5秒/步
> - **成本高**：每次完整浏览约消耗¥0.1-0.5 API费用
> - **仅作为兜底**：常规方法能解决时，绝不使用 browser-use
> - **验证码问题**：遇到验证码时 browser-use 同样会卡住，此时应提示用户手动完成
> - **优先使用 agent-browser**：`agent-browser --cdp 9222` 更快、更便宜、更稳定

**browser-use 最佳实践**：

1. **不要用于登录流程**：让用户手动登录，browser-use 只在登录后操作
2. **限制操作步数**：设置 `max_steps=5`，避免无限循环和成本失控
3. **结果验证**：输出后用JSON Schema校验，确保数据质量
4. **截图 + VLM 分析**：让 browser-use 截图后由 VLM 理解页面，而不是依赖固定选择器

---

## 错误处理原则

1. **快速失败** — 单个数据源3次失败后立即降级，不要反复重试
2. **透明告知** — 每次降级都告诉用户，不偷偷降级
3. **标注精度** — 降级采集的数据在报告中标注来源和精度等级
4. **部分数据也输出** — 宁可用80%的数据出报告，也不等100%数据卡住流程

---

## 通用降级链（总结）

```
首选方案失败 → 同级备选方案（1次重试）→ 下一级路线 → 兜底方案 → 告知用户并标注"未检索到"
```

**绝不做的事：**
- ❌ 同一个失败方案重试超过2次
- ❌ 静默跳过关键数据不告知用户
- ❌ 用猜测数据填充缺失字段
